# 游戏名称：Tupac Amaru: Rise of the Andes

## 一、项目概述
本项目是一款单人策略类电子游戏，玩家将扮演图帕克·阿马鲁二世，在18世纪安第斯地区组织并领导起义，对抗西班牙殖民统治。游戏以节点式地图为基础，通过资源管理、人口动员和据点占领来模拟起义的发展过程。游戏旨在通过简化的策略系统，让玩家理解安第斯社会中“垂直群岛”资源结构以及殖民压迫对社会运作的影响。

## 二、核心玩法
游戏采用回合制结构。每一回合中，玩家可以进行有限数量的操作，主要包括军队调度、据点占领和人口招募。回合结束后，系统自动结算战斗结果、资源变化以及敌方行动。游戏整体操作仅需鼠标点击完成，以保证易上手和清晰表达。

## 三、地图与节点系统

### 3.1 节点类型

游戏地图由若干村落节点构成，节点之间通过路径连接。

#### 3.1.1 村落 (Village)
每个村落既拥有人口资源，也拥有物质资源。根据海拔分为三种类型：
- **高地村落 (Highland)**：产出土豆和羊驼
- **中地村落 (Midland)**：产出玉米和藜麦
- **低地村落 (Lowland)**：产出古柯

### 3.2 村落属性

每个村落拥有以下属性：

#### 3.2.1 基本属性
- **location_name** - 村落名称
- **altitude** - 海拔类型（HIGH / MIDDLE / LOW）
- **control_by_player** - 是否被玩家控制

#### 3.2.2 人口属性
- **population** - 当前人口数量
- 占领后可将人口转化为军队单位

#### 3.2.3 资源属性
- **produced_resource_types** - 该村落生成的资源种类
- **resources** - 拥有的资源存量 {资源类型: 数量}
- 资源可以存储、交易和运输

#### 3.2.4 军队属性
- **stationed_units** - 驻守的军队单位列表

### 3.3 镜头与节点分配

### 3.5 人口与资源生成

村落的资源生成速度与其人口数量成正比，遵循以下计算公式：

**资源生成速度 = 人口数量 ÷ 50 （资源/回合）**

例如：
- Cusco（人口300）：300 ÷ 50 = 6 资源/回合
- Tinta（人口150）：150 ÷ 50 = 3 资源/回合
- Pilcopata（人口60）：60 ÷ 50 = 1.2 资源/回合

生成的资源类型由村落的海拔和初始配置决定。

### 3.6 人口招募系统

玩家可以在控制的村落中将人口转化为军队单位。

**招募规则**：
- **成本**：25人口 → 1个军队单位
- **单位类型概率**：
  - 20% 概率：**Female Corps (女性营)**
  - 80% 概率：**Rebel Army (起义军)**

每次招募是独立的随机结果，遵循上述概率分布。

### 3.7 节点控制与占领

游戏开始时，系统自动占领 Tinta 村落，使其成为唯一被玩家控制的村落。除 Tinta 外，其余所有村落都处于殖民体系控制之下。玩家需要通过军队占领逐步转化其他村落为起义军控制。只有被玩家控制的村落才能进行人口动员、资源存储和部队驻扎。

# 四、资源系统

## 4.1 资源类型与效果

### 高地资源
- **土豆 (Potato)**: 填充饱腹值 +50（提供给起义军）
- **羊驼 (Llama)**: 资源运输速度 ×2（提供给单位）

### 中地资源
- **玉米 (Corn)**: 填充饱腹值 +30，军队战斗力 ×1.2（提供给起义军）
- **藜麦 (Quinoa)**: 
  - 提供给单位：填充饱腹值 +20，治疗 +20，移速 ×1.2，持续3回合
  - 提供给村庄：资源生产速度 ×1.2，持续3回合

### 低地资源
- **古柯 (Coca)**: 治疗单位 +50（提供给单位）

## 4.2 资源存储系统

### 村庄存储
- 每个村庄有存储上限
- 村庄原生产的资源上限为 10 个
- 从其他地方运来的资源不受原生产资源上限影响，各自上限 10 个
- 例：产土豆+羊驼的村庄可存储 10 土豆 + 10 羊驼 + 10 玉米 + 10 古柯

### 单位存储
- 每个单位总容量为 5 个资源
- 资源种类与数量可混合
- 例：3 土豆 + 2 玉米，或 5 古柯

## 4.3 资源运输系统

### 运输时间
- **同一镜头内**：2 回合
- **不同镜头间**：仅按跨越镜头数 × 4 回合（无基础值）
- 例：跨越两个镜头需 2×4 = 8 回合

## 4.4 村庄资源消耗系统

### 4.4.1 可消耗资源

村庄仅可以自动消耗以下三种资源：
- **土豆 (Potato)**
- **玉米 (Corn)**
- **藜麦 (Quinoa)**

古柯 (Coca) 和羊驼 (Llama) 不被消耗。

### 4.4.2 消耗条件

村庄自动消耗食物需要满足以下条件：
1. 村庄处于**饥饿状态** (hunger_status = true)
2. 村庄拥有上述可消耗资源之一

### 4.4.3 消耗速率与累积

**消耗速率公式**：
```
每回合消耗值增加 = 村庄人口 × 0.01
```

例如：
- Cusco（人口300）：每回合消耗值 +3.0
- Tinta（人口150）：每回合消耗值 +1.5
- Pilcopata（人口60）：每回合消耗值 +0.6

系统维护每个村庄的**累积消耗值** (consumption_accumulator)，每回合都要计算并更新。

### 4.4.4 消耗优先度

当村庄需要消耗食物时，遵循以下优先度顺序：
1. **土豆 (Potato)** - 首选
2. **玉米 (Corn)** - 次选
3. **藜麦 (Quinoa)** - 最后选择

同等优先度资源数量足足，优先消耗数量最多的资源。

### 4.4.5 消耗与饥饿状态切换

**每个非玩家回合（系统计算环节）的执行流程**：

```
1. 计算消耗值
   累积消耗值 += 人口 × 0.01

2. 检查是否需要消耗（消耗值 ≥ 1）
   if 累积消耗值 >= 1.0:
     设置村庄为饥饿状态 (hunger_status = true)
     资源生成速率倍数 = 0.2
   else:
     if 村庄有可消耗的食物:
       设置村庄为非饥饿状态 (hunger_status = false)
       资源生成速率倍数 = 1.0

3. 消耗食物（当饥饿状态为true且有食物）
   while 累积消耗值 >= 1.0 AND 村庄有可消耗食物:
     按优先度消耗一个食物
     累积消耗值 -= 1.0

4. 更新饥饿状态
   if 消耗后 没有可消耗的食物:
     设置为饥饿状态 (hunger_status = true)
     资源生成速率倍数 = 0.2
   else:
     设置为非饥饿状态 (hunger_status = false)
     资源生成速率倍数 = 1.0
```

### 4.4.6 示例场景

**场景1：Tinta 有足够食物**
```
初始状态：
- 人口：150
- 资源：2个土豆，0个玉米，0个藜麦
- 累积消耗值：0.2
- 饥饿状态：false
- 资源生成速率倍数：1.0

非玩家回合执行：
1. 计算消耗值：0.2 + 150×0.01 = 0.2 + 1.5 = 1.7
2. 1.7 ≥ 1.0，设置为饥饿状态，倍数 = 0.2
3. 消耗食物循环：
   - 第一次：消耗1个土豆，消耗值 = 1.7 - 1 = 0.7
   - 0.7 < 1.0，退出循环
4. 村庄仍有食物，设置为非饥饿状态，倍数 = 1.0

最终状态：
- 资源：1个土豆，0个玉米，0个藜麦
- 累积消耗值：0.7
- 饥饿状态：false
- 资源生成速率倍数：1.0
```

**场景2：Tinta 没有食物**
```
初始状态：
- 人口：150
- 资源：0个土豆，0个玉米，0个藜麦
- 累积消耗值：0.2
- 饥饿状态：false
- 资源生成速率倍数：1.0

非玩家回合执行：
1. 计算消耗值：0.2 + 150×0.01 = 0.2 + 1.5 = 1.7
2. 1.7 ≥ 1.0，设置为饥饿状态，倍数 = 0.2
3. 消耗食物循环：
   - 检查是否有可消耗食物：没有
   - 退出循环
4. 村庄没有可消耗食物，保持饥饿状态，倍数 = 0.2

最终状态：
- 资源：0个土豆，0个玉米，0个藜麦
- 累积消耗值：1.7
- 饥饿状态：true
- 资源生成速率倍数：0.2
```

## 4.5 资源影响机制

高地资源不足时，人口会逐渐减少；中地资源不足时，军队规模会下降；低地资源不足时，单位移动速度或行动效率会降低。玩家需要同时控制不同海拔的资源点，以维持起义的稳定发展。



# 五、单位与战斗系统

## 5.1 单位类型

游戏中有两种可操控单位类型，分别实现为 Unit 类的子类：

### 5.1.1 Rebel Army (起义军) [RebelArmy.gd]
- **特性**：进攻型单位
- **生命值**：100
- **饱腹值**：100
- **攻击力**：30
- **驻扎状态攻击力倍数**：×1（无倍数）

### 5.1.2 Female Corps (女性营) [FemaleCorps.gd]
- **特性**：防守型单位
- **生命值**：80
- **饱腹值**：100
- **基础攻击力**：20
- **驻扎状态攻击力倍数**：×2（驻扎时攻击力变为40）

## 5.2 单位属性与状态系统

### 5.2.0 单位基础属性

每个单位拥有以下基础属性：

#### 位置与归属
- **current_node** - 当前所在村落
  - Moving 状态：保持为出发时的村落，直到单位到达目标村落
  - Attacking 或 Stationed 状态：为该单位所在的村落
- **origin_village** - 单位原始遵属的村落

#### 倍数属性（均默认值为 1.0）
- **combat_multiplier** - 战斗力倍数（装备玉米时 ×1.2）
- **movement_speed_multiplier** - 移动速度倍数（装备藜麦时 ×1.2）
- **transport_speed_multiplier** - 运输速度倍数（装备羊驼时 ×2.0）

#### 饱腹值消耗
- **get_current_satiety_consumption()** - 根据当前状态返回每回合的饱腹值消耗值
  - Stationed 状态：返回 -10
  - Moving 或 Attacking 状态：返回 -15

### 5.2.1 Moving (移动)
- 单位正在移动中
- 饱腹值消耗：每回合 -15（通过 get_current_satiety_consumption() 返回）
- **current_node**：保持为出发时的村落，直到单位到达目标村落
- 当单位到达目的地村落时，current_node 更新为目标村落，状态自动转变为 Stationed

### 5.2.2 Attacking (进攻)
- 单位不在移动，且位置在未被玩家控制的村庄
- 饱腹值消耗：每回合 -15（通过 get_current_satiety_consumption() 返回）
- **current_node**：当前进攻所在的村落
- 参与战斗，生命值会受到敌方单位伤害

### 5.2.3 Stationed (驻扎)
- 单位不在移动，且位置在已被玩家控制的村庄
- 饱腹值消耗：每回合 -10（通过 get_current_satiety_consumption() 返回）
- **current_node**：驻扎所在的村落
- 驻守单位本身处于非战斗状态，Female Corps 在驻守时获得 ×2 攻击力倍数

### 5.2.4 控制状态与单位状态转换规则
- 当单位不在移动时：
  - 若所在节点 control_by_player = true，单位状态为 **Stationed**
  - 若所在节点 control_by_player = false，单位状态为 **Attacking**
- 当敌人入侵玩家控制节点时，该节点立即变为 uncontrolled，该节点内原驻守单位立即从 **Stationed** 转为 **Attacking** 并参与战斗。

## 5.3 单位生命周期

### 5.3.1 饱腹值与生命值消耗规则

**消耗阶段**
- **第一阶段**：饱腹值 > 0 时，每回合扣除饱腹值
  - Stationed 状态：-10 饱腹值
  - Moving 或 Attacking 状态：-15 饱腹值

- **第二阶段**：饱腹值 ≤ 0 时，开始扣除生命值
  - 每回合生命值下降，下降数值与当前状态下的饱腹值消耗相同
  - Stationed 状态：-10 生命值/回合
  - Moving 或 Attacking 状态：-15 生命值/回合

### 5.3.2 单位死亡条件

- **仅在生命值 ≤ 0 时死亡**
- 饱腹值 ≤ 0 不直接导致死亡，仅转入生命值消耗阶段

### 5.3.3 生命值恢复
- Rebel Army 初始值：100
- Female Corps 初始值：80
- 恢复方式：仅通过资源恢复（Coca 资源 +50）

### 5.3.4 饱腹值对战斗力的影响
- **正常状态**（饱腹值 ≥ 20）：单位维持基础攻击力
- **低饱腹值状态**（饱腹值 < 20）：单位攻击力变为原先的一半（向下取整）
  - 例：Rebel Army 攻击力从 30 变为 15
  - 例：Female Corps 驻扎时攻击力从 40 变为 20

## 5.4 战斗系统

### 5.4.1 敌方单位属性
- **生命值**：80
- **攻击力**：25
- **饱腹值**：无（不需玩家维持）
- **特点**：无特殊状态或倍数

### 5.4.2 战斗结算规则

**多单位参战场景**：当多个我方单位与多个敌方单位在同一回合内交战时，遵循以下规则：

1. **伤害计算**
   - **当前攻击力确定**：根据单位的饱腹值状态计算
     - 饱腹值 ≥ 20：使用基础攻击力
     - 饱腹值 < 20：使用基础攻击力的一半（向下取整）
   
   - 敌方总伤害 = 敌方单位数量 × 25
   - 我方每单位受伤 = ⌊敌方总伤害 ÷ 我方单位数量⌋ （向下取整）
   
    - 我方总伤害 = 各参战单位当前攻击力之和（含倍数和饱腹值削弱）
   - 敌方每单位受伤 = ⌊我方总伤害 ÷ 敌方单位数量⌋ （向下取整）

2. **回合末结算**
   - 所有伤害在回合末统一结算
   - 若双方单位在同一回合互相攻击导致都降至 0 血量，则**同时死亡**

3. **结算顺序（同一回合）**
    - 先进行战斗伤害计算与结算
    - 后进行饱腹值扣减（以及饱腹值耗尽后的生命值扣减）

4. **示例**
   ```
   我方：3个Rebel Army单位，各基础攻击力30，饱腹值≥20
   敌方：2个单位，各攻击力25
   
   我方总伤害 = 3 × 30 = 90
   敌方每单位受伤 = ⌊90 ÷ 2⌋ = 45
   
   敌方总伤害 = 2 × 25 = 50
   我方每单位受伤 = ⌊50 ÷ 3⌋ = 16
   
   ---
   
   变量场景：我方1个Rebel Army单位，饱腹值 < 20（攻击力削弱至15）
   1个女性营驻扎状态，饱腹值 ≥ 20（攻击力40）
   对方3个单位，各攻击力25
   
   我方总伤害 = (15 + 40) = 55
   敌方每单位受伤 = ⌊55 ÷ 3⌋ = 18
   
   敌方总伤害 = 3 × 25 = 75
   我方Rebel Army受伤 = ⌊75 ÷ 2⌋ = 37
   我方Female Corps受伤 = ⌊75 ÷ 2⌋ = 37
   ```

### 5.4.3 战斗参与条件
- **Attacking 状态**：单位会自动参与战斗
- **Stationed 状态**：单位默认不参与战斗；若节点因入侵变为 uncontrolled，该单位立即转为 **Attacking** 后参与战斗
- **Moving 状态**：单位不参与战斗

## 5.5 单位与战场

玩家可以将单位移动至其他村落，占领未控制的村落或防守已有据点。移动时间由两个村落所属的镜头决定（详见4.3资源运输系统中的镜头间距离规则）。战斗系统为自动结算，所有伤害计算均采用向下取整，不存在小数伤害。

# 六、殖民干扰系统
游戏中设有殖民势力的反制机制。系统会定期触发干扰事件，例如减少某一资源点的产出、削弱某个村落的人口，或对玩家控制的节点发起进攻。这些事件用于模拟殖民政府的征税、劳役制度以及军事镇压。

# 七、胜负条件
玩家的目标是逐步扩大起义势力范围。游戏胜利条件为占领所有资源点与村落，或在资源系统稳定的情况下坚持一定回合数。若玩家因资源崩溃导致人口或军队无法维持，则判定失败。

# 八、界面与交互
游戏采用简洁界面设计。玩家通过点击节点查看信息，并选择移动、占领或招募等操作。系统通过文字提示告知玩家当前资源状态、战斗结果以及敌方行动，帮助理解游戏逻辑。

# 九、设计目标
本游戏的设计重点在于将历史概念转化为可操作的游戏机制。通过资源依赖关系和节点控制，玩家可以直观理解安第斯地区的生态结构及其对社会组织的影响。同时，游戏通过模拟殖民干扰与资源压力，使玩家体验起义在现实中的困难与限制。

# 十、开发范围与实现策略
本项目为单人开发，开发周期约为一个月。设计上优先保证核心玩法的完整性与可运行性，避免复杂系统。最小可实现版本包括基础地图、节点占领、资源影响和自动战斗系统。在此基础上逐步完善界面和提示信息，以提升可玩性与表达效果。
